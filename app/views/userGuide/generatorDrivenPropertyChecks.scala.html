Generator-driven property checks

Generator-driven property checks in ScalaTest use generators from org.scalatest.prop. Mix in GeneratorDrivenPropertyChecks to get forAll, whenever, and the common generator helpers.

If you are also using table-driven checks, you can mix in PropertyChecks, which combines generator-driven and table-driven APIs.

Core APIs

  • GeneratorDrivenPropertyChecks: forAll entry points and property syntax.
  • Generator[A]: typed generators with edge-value and shrinking support.
  • CommonGenerators: ready-made generators and combinators.
  • Configuration: tuning controls such as minSuccessful and sizeRange.

Getting started

The simplest form of forAll relies on implicit generators for parameter types: 

import org.scalatest.prop.GeneratorDrivenPropertyChecks._

import org.scalatest.matchers.should.Matchers._


forAll { (s: String) =>
  (s + s).length shouldEqual s.length * 2
}

By default, ScalaTest checks each property against a number of generated values (default minSuccessful is 10).

Supplying explicit generators

Default generators are broad. For focused properties, supply explicit generators: 

import org.scalatest.prop.CommonGenerators.{strings, posZIntsBetween}


forAll(strings, posZIntsBetween(0, 1000)) { (s: String, n) =>

  (s * n.value).length shouldEqual s.length * n.value

}

Range constructors such as intsBetween, longsBetween, posIntsBetween, and posZIntsBetween are often the best first step to reduce noise and improve failure diagnostics.

Custom generators

You can build custom generators with for-comprehensions via map, flatMap, and filter: 

import org.scalatest.prop.{Generator, CommonGenerators}

import CommonGenerators.posInts



case class Rectangle(w: Int, h: Int)



implicit val rectangleGen: Generator[Rectangle] =

  for {

    w <- posInts

    h <- posInts

  } yield Rectangle(w.value, h.value)



forAll { (r: Rectangle) =>

  r.w * r.h should be > 0

}

Supplying argument names

You can provide argument names to improve failure output: 

forAll ("a", "b") { (a: String, b: String) =>
  a.length + b.length shouldEqual (a + b).length
}

Failure messages will report named arguments instead of arg0, arg1, and so on.

Supplying both generators and argument names

You can provide generator/name pairs together: 

import org.scalatest.prop.CommonGenerators.{intsBetween, posIntsBetween}


val numerators = intsBetween(-1000, 1000)

val denominators = posIntsBetween(1, 1000)


forAll((numerators, "n"), (denominators, "d")) { (n: Int, d) =>

  n / d.value * d.value + (n % d.value) shouldEqual n

}

Using whenever and discarded evaluations

Use whenever when a property has explicit preconditions. Evaluations that do not satisfy the precondition are discarded rather than failed. 

forAll { (n: Int, d: Int) =>

  whenever(d != 0) {

    (n / d) * d + (n % d) shouldEqual n

  }

}

If too many evaluations are discarded, the run is considered exhausted. In practice, the best fix is usually to tighten the generators so valid inputs are produced more often.

Property check configuration

Property checks can be tuned via supertrait Configuration. The key parameters are:

Configuration Parameter Default Value Meaning
minSuccessful 10 minimum successful evaluations required for a pass
maxDiscardedFactor 5.0 allowed discarded evaluations, as a factor of successful ones
minSize 0 minimum size parameter for size-sensitive generators
sizeRange 100 range added above minSize to compute max generation size
workers 1 number of worker threads to use during evaluation

You can override defaults for a scope with an implicit PropertyCheckConfiguration: 

import org.scalatest.prop.Configuration.PropertyCheckConfiguration


// Use 'override' when your suite mixes in a trait that already defines generatorDrivenConfig.

implicit override val generatorDrivenConfig: PropertyCheckConfiguration =
  PropertyCheckConfiguration(minSize = 10, sizeRange = 20)

You can override config for one forAll call with config params: 

forAll (minSuccessful(500), maxDiscardedFactor(2.0)) { (n: Int) =>

  n + 0 shouldEqual n

}

If you are already passing names or generators, place config params after those arguments.

When failures occur, ScalaTest reports generated arguments and an initial random seed. Re-running with the same seed helps reproduce the failure deterministically.

For more information, see Configuration and PropertyCheckResult.
